# discussions/ — high-level design discussions

This directory holds the **"what should we build, and why"** reasoning behind jidoka — the high-level discussions you have with an agent, written down so they aren't re-litigated. Each doc is **living current-best thinking**, edited in place as the thinking matures.

It is deliberately **not** a wiki and **not** a frozen archive:

- Unlike a wiki (neutral reference — "how it works"), a discussion is **argued**: a question, the options weighed, and the call.
- Unlike `../exec-plans/completed/` (frozen records), a discussion **evolves** — you edit it as the thinking moves, rather than freezing it.

It is **not governed by the plan-lifecycle convention** — `../CONVENTION.md` covers only the `exec-plans/` work funnel. Where a repo keeps this kind of doc (here, `docs/discussions/`) is its own choice.

## When to write here

When you and the user hash out high-level direction — should we build X, which approach, why — **the conclusion is captured here as one doc.** If the resolution also implies work to do, that spawns a `../exec-plans/backlog/` item (or an active plan); the *reasoning* stays here. A discussion whose answer is "decided, nothing to build" lives here and only here.

## Doc shape

- **One file per topic**, topic-named (`cli-over-mcp.md`, `review-pipeline.md`) — referenced by subject, not date.
- **The H1 carries the one-line stance** — `# <Topic> — <current stance>` (e.g. `# CLI over MCP — the CLI binary is already the agent-agnostic interface; defer MCP to post-v1`). This puts the conclusion at the very top, with the discussion below, so a reader gets the answer immediately. It is also exactly what the catalog re-derives (below), so keep it a faithful one-line summary of where the doc currently lands.
- **Reference, don't paste** (same rule as `../CONVENTION.md` rule 2): capture rationale and point at code by `path:symbol` rather than pasting snippets that go stale.

## Lifecycle

A discussion **evolves in place** while it is current truth — grep it freely; it is true *right now*. When a decision is **reversed**, it does not linger here as a stale graveyard entry: `git mv` it to `../exec-plans/completed/<slug>.md` and prepend the superseded stamp —

```text
> STATUS: superseded · <YYYY-MM> · replaced-by <what>, kept as record
```

— so `completed/` (the single home for frozen history) keeps the record and `discussions/` stays purely current.

## Catalog (`index.md`)

`index.md` is a **re-derivable catalog**, not hand-maintained meta — one line per doc, its H1 stance, regenerated by grep so it can't drift:

```sh
find docs/discussions -name '*.md' ! -name 'index.md' ! -name 'AGENTS.md' -exec grep -hm1 '^# ' {} + | sed 's/^# /- /' | sort
```

(The `! -name` filters skip this file and the catalog itself, and `find` doesn't error on an empty directory; `| sort` makes the order deterministic.)

## Cross-references

- `../CONVENTION.md` — the portable plan-lifecycle convention (covers `exec-plans/` only; discussions are out of scope).
- `../exec-plans/AGENTS.md` — where reversed decisions land (`completed/`) and the provenance stamp.
- `../exec-plans/completed/` — the single home for frozen history, including superseded discussions.
