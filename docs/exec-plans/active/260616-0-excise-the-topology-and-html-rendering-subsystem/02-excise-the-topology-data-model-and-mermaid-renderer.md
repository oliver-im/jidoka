# Unit 02 — Excise the topology data model and Mermaid renderer
**Blocked by:** 01-remove-the-html-browser-output-and-topology-cli-command**Agents involved:** main only**Topology:** none
## Summary

With the view surface gone, remove the topology data model itself — the `Topology`/`Agent`/`Output` types, their zod schemas and serializers, topology parsing, and the `topology` field on `Unit` — and delete the Mermaid renderer (`mermaid.ts` + its `graph.ts` helper). Plan-markdown parsing and plan/unit validation stay fully intact.

### Tasks
- `ts/types.ts`: remove `Output` + `outputSchema` + `serializeOutput`, `serializeAgent`, `serializeTopology`, `modelSchema`/`Model`, `executionModeSchema`/`ExecutionMode`, `Agent` + `baseAgentSchema` + `agentSchema`, `Topology` + `topologySchema`, and `parseTopology` + `parseTopologyJson`. Remove the `topology` field from the `Unit` interface and from `unitSchema`. Keep the `ReviewStep` machinery, `Unit` (minus topology), `Plan`/`planSchema`, `ParseResult`, `formatZodError`, `parsePlan`, `parsePlanJson`, and `stripBom`.
- `ts/validate.ts`: remove the `escapeId` import from `./mermaid.js`; delete `validateTopology` and its helpers `validateTopologyInto`, `collectAndValidateIds`, `validateAgentFields`, `validateDependenciesInScope`, `detectAgentCycles`; remove the agent/topology `ValidationError` kinds and their `formatError` cases, including the `mermaid_id_collision` check; remove the `unit.topology` validation loop inside `validatePlan`; drop the `Topology`/`Agent` type imports. KEEP `isValidId` + `AGENT_ID_RE` (used by `hook.ts:isValidSessionId`), `isValidSlug`, `isValidUnitId`, `validatePlan`, `detectUnitCycles`, the `Color` type, and the `empty_task_summary` kind (`validatePlan` emits it for the plan at `validate.ts:121`, not just for topology).
- `ts/parse-markdown.ts`: drop the `Topology` type import and the `parseTopologyJson` import; delete `extractTopologyFence` and `FENCE_LINE_RE`; in `parsePlanMarkdown` set `body_markdown` straight from the section text and drop the `unit.topology` assignment. A fenced block whose info string is `topology` now simply stays inline in the body as ordinary prose. Update the function docstring to drop the topology-fence description.
- `ts/render-md.ts`: drop the `mermaid` import and the `Topology` type import; delete `unitTopologyBlock`; remove the `topologyLabel` and `topologyBlock` locals from `buildUnitMd` and the corresponding eta template variables.
- `templates/unit.md.eta`: remove the `**Topology:**` line and the `topologyBlock` output slot.
- Delete modules: `ts/mermaid.ts`, `ts/graph.ts`.
- Tests: delete `ts/__tests__/mermaid.test.ts`, `graph.test.ts`. Trim all topology/agent cases from `types.test.ts`, `validate.test.ts`, `parse-markdown.test.ts` (large topology-fence surface), and `render-md.test.ts`. Delete the topology JSON fixtures `valid_minimal.json`, `valid_nested.json`, `valid_team.json`, `valid_plan_with_topology.json`, and the agent/topology `invalid_*` ones (`invalid_bad_id`, `invalid_blocked_by_not_found`, `invalid_cycle`, `invalid_duplicate_id`, `invalid_empty_agents`, `invalid_empty_nested_agents`, `invalid_empty_role`, `invalid_empty_task_summary`, `invalid_self_dep`), and remove any lingering references to them or to the deleted `topology` field across the suite. Keep the plan fixtures (`valid_plan_minimal`, `valid_plan_sequential`, `invalid_plan_*`, `plan.md`, `bad.md`).

### Acceptance
- No reference to `Topology`, `Agent`, or a unit `topology` field remains anywhere under `ts/` (source or tests).
- `npm run build && npm test && npm run typecheck` all green; plan-markdown parsing and plan/unit validation behave exactly as before for non-topology input.
- Rebuild and commit `dist/cli.js` so the committed bundle matches source.
